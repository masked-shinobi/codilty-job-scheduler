# Design Decisions — JobFlow Distributed Job Scheduler

## 1. Why Postgres-as-Queue Instead of Redis/RabbitMQ

**Decision**: Use PostgreSQL as both the data store and the queue backing store.

**Rationale**:
- **Operational simplicity**: One database to manage, back up, and monitor. No separate message broker infrastructure.
- **Transactional consistency**: Job state transitions are atomic — a job cannot be created/claimed/completed without the corresponding metadata being updated in the same transaction.
- **`SELECT FOR UPDATE SKIP LOCKED`**: Postgres provides native row-level locking that gives us exactly the semantics we need for atomic job claiming. `SKIP LOCKED` ensures that concurrent workers never block each other — they simply skip already-locked rows and move to the next eligible job.
- **Cost**: No additional service to pay for or manage on free tiers. Redis/RabbitMQ would require a separate hosted instance.
- **Durability**: Jobs survive database restarts. No risk of losing messages in a volatile in-memory queue.

**Trade-off**: At very high throughput (>10k jobs/second), a dedicated message broker like Redis would outperform Postgres. For the workloads this system targets, Postgres is more than sufficient.

## 2. Why UUID Primary Keys

**Decision**: All entities use UUID v4 primary keys.

**Rationale**:
- **No enumeration attacks**: Sequential integer IDs leak information (e.g., total job count).
- **Distributed generation**: UUIDs can be generated by any service without coordination — important when multiple workers create execution records concurrently.
- **Merge-safe**: If we ever need to merge data from multiple environments, UUIDs won't collide.

**Trade-off**: UUIDs are larger (16 bytes vs 4/8 bytes for integers) and can fragment B-tree indexes. For our scale, this is negligible.

## 3. Cascade Delete Rationale

| Parent → Child | Cascade? | Justification |
|---|---|---|
| Organization → Project | Yes | Deleting an org means all its projects are no longer needed |
| Project → Queue | Yes | Queues are scoped to a project |
| Queue → Job | Yes | Jobs are meaningless without their queue |
| Job → JobExecution | Yes | Execution history is tied to the job |
| Job → JobLog | Yes | Logs are tied to the job |
| Worker → JobExecution | **SetNull** | Execution history must survive worker death for audit purposes |
| Worker → WorkerHeartbeat | Yes | Heartbeats are operational data, not audit data |
| Organization → OrgMember | Yes | Membership is scoped to the org |

**Key trade-off**: `Worker → JobExecution` uses `SetNull` instead of `Cascade` because execution history is an audit trail. If a worker is cleaned up, we still want to see that "Worker X ran Job Y at time Z and it failed." The `workerId` becomes null but the record persists.

## 4. Why SKIP LOCKED Guarantees Atomicity

The `SELECT FOR UPDATE SKIP LOCKED` pattern works as follows:
1. Worker A issues `SELECT id FROM Job WHERE status = 'QUEUED' ... FOR UPDATE SKIP LOCKED LIMIT 1`
2. Postgres locks the selected row — no other transaction can read or modify it via `FOR UPDATE`.
3. Worker B issues the same query concurrently. The `SKIP LOCKED` clause causes Postgres to skip the row locked by Worker A and return the *next* eligible row.
4. Both workers proceed without blocking each other, and no job is ever claimed twice.

This is superior to:
- **Advisory locks**: More complex, harder to debug, no automatic cleanup.
- **Application-level locking**: Race conditions, distributed coordination complexity.
- **Optimistic locking (version columns)**: Requires retry loops, wastes queries.

## 5. Indexing Rationale for the Claim Query

The critical claim query is:
```sql
SELECT id FROM "Job"
WHERE "queueId" = $1 AND status = 'QUEUED' AND "runAt" <= now()
ORDER BY priority DESC, "runAt" ASC
FOR UPDATE SKIP LOCKED LIMIT 1
```

**Composite index**: `(queueId, status, runAt)` — this allows Postgres to:
1. Seek directly to the correct queue.
2. Filter by status within that queue.
3. Scan by `runAt` to find eligible jobs.

This avoids a sequential scan on the potentially large `Job` table.

## 6. Polling vs WebSockets for Dashboard Updates

**Decision**: Use React Query with `refetchInterval: 5000` (polling).

**Rationale**:
- **Simplicity**: No WebSocket server, no connection management, no reconnection logic.
- **Stateless**: The API server remains stateless and horizontally scalable. WebSockets introduce statefulness.
- **Sufficient for dashboards**: A 5-second refresh interval provides near-real-time updates for monitoring use cases without the complexity of WebSockets.
- **Free tier friendly**: WebSocket connections count against connection limits on free hosting tiers.

**Trade-off**: For truly real-time use cases (sub-second updates), WebSockets would be superior. This can be added later as an optional enhancement.

## 7. RBAC Design

**Decision**: Simple role-based access control with three roles: OWNER, ADMIN, MEMBER.

| Action | OWNER | ADMIN | MEMBER |
|---|---|---|---|
| Read org/projects/queues/jobs | ✅ | ✅ | ✅ |
| Create projects/queues/jobs | ✅ | ✅ | ✅ |
| Pause/resume queues | ✅ | ✅ | ❌ |
| Delete projects/queues | ✅ | ✅ | ❌ |
| Manage org members | ✅ | ✅ | ❌ |
| Delete organization | ✅ | ❌ | ❌ |

Enforcement happens via middleware that resolves the organization context from the request and checks the user's `OrgMember.role`.

## 8. Queue Health Formula

| Status | Criteria |
|---|---|
| HEALTHY | Success rate ≥ 90% AND backlog < 50 jobs |
| DEGRADED | Success rate 70–90% OR backlog 50–200 jobs |
| UNHEALTHY | Success rate < 70% OR backlog > 200 jobs |

Success rate is calculated over the last 24 hours. Backlog is the current count of `QUEUED` status jobs.
